## Contribution Path

The license explicitly allows: "modifying code for the purpose of contributing bug fixes or improvements back to Remotion." We're clear to fork and PR.

## Where RE_DITHER Fits

### Option A: New transition presentation (lowest friction)

**Location:** `packages/transitions/src/presentations/`

Existing transitions: fade, flip, slide, wipe, clock-wipe, iris, none.

**Pattern:** Each transition is:
1. A props type (`DitherProps`)
2. A component receiving `TransitionPresentationComponentProps<DitherProps>`
3. A factory function returning `{component, props}`

The component uses `presentationProgress` (0-1) to drive the effect.

**Our contribution: `dither.tsx`**
```typescript
export type DitherProps = {
  aggression?: number;        // 0-100, default 50
  tint?: 'none' | 'denim' | 'gold' | 'ember' | 'moss' | 'denim-gold';
  mode?: 'materialize' | 'dissolve';
};
```

Uses Canvas API offscreen to apply Floyd-Steinberg + white-destroyer to the entering/exiting scene at the current progress point. This is novel — no other transition does pixel-level dithering.

### Option B: Standalone package `@remotion/dither` (medium friction)

**Model:** `packages/noise/` — standalone utility package.

Exports:
- `useDither(src, options)` — hook that returns a dithered frame
- `useWhiteDestroyer(src, aggression, tint)` — hook for jeans texture
- `<DitherImage>` — component wrapping `<Img>` with real-time dithering
- `floydSteinberg(imageData)` — raw utility

### Option C: Both (recommended)

Ship the transition first (Option A) — it's a single file PR, matches existing patterns, gets merged fast. Then propose the standalone package as a follow-up.

## Key Technical Considerations

1. **Canvas in Remotion:** Remotion renders in a headless browser (Puppeteer/Playwright). Canvas API works fine. The `noise` package already does heavy per-frame computation.

2. **Performance:** Floyd-Steinberg is O(w*h) per frame. At 1080p, that's ~2M pixels. With caching (same as our renderer.ts), this is fine for offline rendering. For `<Player>` (real-time preview), may need resolution downscaling.

3. **Determinism:** Same `presentationProgress` value must produce identical output. Our dither engine is deterministic (no randomness in Floyd-Steinberg itself).

4. **SSR compatibility:** The transition component needs to work in Remotion's SSR pipeline. Canvas API is available in the rendering environment.
